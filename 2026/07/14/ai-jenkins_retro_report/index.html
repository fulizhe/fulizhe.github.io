Jenkins zIsTimerTrigger 复盘：当运行时结构假设毁了整个下午

2026-07-14T03:30:00.000Z2026-07-23T14:23:57.571ZChina

这篇复盘的起点，来自一次没做到位的对话。

背景

本轮问题发生在 Jenkins Shared Library 变量 jsl/vars/zIsTimerTrigger.groovy 中，目标是判断构建链路是否由定时器触发。

用户的真实诉求不是只看当前构建的直接原因，而是递归追溯最上游根因：

  • 如果当前构建是被上游任务触发的
  • 但上游任务最初是由 TimerTrigger 触发的
  • 那么 zIsTimerTrigger() 也必须返回 true

典型场景：

  • 当前 Job：Started by upstream project "pt2_test" build number 545
  • 上游 Job：originally caused by: Started by timer

在这个场景里，期望返回值应为 true。也就是说，判断逻辑需要递归追溯整条上游链路的最上游根因，而不是仅判断当前构建的直接 cause。

问题现象

在实际运行中，函数多次出现以下问题：

  1. 直接访问 currentBuild.rawBuild 报错
  2. 错误假设 getBuildCauses() 的返回值结构固定
  3. 对 _class 的直接访问在某些 cause 对象上失败
  4. 对 upstreamCauses 的递归假设与真实 Jenkins 返回结构不一致
  5. 最终导致函数在”明明上游根因是 timer”的情况下仍返回 false

随后又出现多个不同的 Groovy 异常：

  • MissingPropertyException: No such property: rawBuild for class: org.jenkinsci.plugins.workflow.job.WorkflowRun
  • MissingPropertyException: No such property: _class for class: hudson.triggers.TimerTrigger$TimerTriggerCause

代码演进简述

第一版：只判断当前构建直接 cause

最初实现只检查当前构建是否直接由 timer 触发：

问题：只能识别”直接 timer 触发”，无法处理 upstream 链路。

第二版：尝试访问 rawBuild

尝试用 currentBuild.rawBuild 继续做递归，但在 Pipeline 场景中，这个属性并不总是可直接访问，出现：

  • MissingPropertyException: No such property: rawBuild for class: org.jenkinsci.plugins.workflow.job.WorkflowRun

问题：入口就不稳。

第三版：改用 getBuildCauses()

改为使用 Jenkins Pipeline 的白名单 API currentBuild.getBuildCauses()，方向是对的。但错误地认为返回值一定是 Map 或 JSON 对象，于是开始读取 cause._class 和 cause.upstreamCauses，结果发现运行时对象并不一致。

第四版：混合递归与对象兼容

试图兼容 JSONObject、普通 cause 对象、上游构建对象，但又在某些分支里直接读取 _class，导致：

  • MissingPropertyException: No such property: _class for class: hudson.triggers.TimerTrigger$TimerTriggerCause

说明对象类型的判断顺序仍然不对。

关键定位

从运行日志可以确认，当前结构里真正可依赖的是：

  • 当前构建能拿到 upstreamProject
  • 当前构建能拿到 upstreamBuild
  • 上游构建确实存在
  • 上游构建的根因确实可能是 TimerTrigger

这说明真正应该依赖的是 BuildUpstreamCause -> upstreamProject/upstreamBuild -> 上游 run -> 再读 causes，而不是去猜测 _class 是否存在。

根因分析

直接根因

最初把 Jenkins Pipeline 的 cause 对象当成了统一格式的数据结构来处理，但实际运行时并非如此。currentBuild.getBuildCauses() 的返回内容可能包含：

  • org.jenkinsci.plugins.workflow.support.steps.build.BuildUpstreamCause
  • hudson.triggers.TimerTrigger$TimerTriggerCause
  • net.sf.json.JSONObject
  • Jenkins 内部的其他 cause 类型对象

这些对象并不都具备 _class 属性，也不都适合直接以 Map 的方式读取。

为什么会失败

在没有完全确认运行时结构的情况下，过早做了以下假设：

  • rawBuild 可直接访问
  • cause._class 一定存在
  • upstreamCauses 一定存在且可递归
  • getBuildCauses() 返回的是统一的 JSON/Map 结构

这些假设都不稳定，导致逻辑不断偏离真实环境。

复盘：为什么这轮没有一次做对

1. 过早假设了 Jenkins 返回结构

一开始把 Jenkins Pipeline 的 cause 结构当成”固定 JSON / 固定 Map”处理，但真实返回中存在多种对象形态。这导致在代码里直接访问 rawBuild、_class、upstreamCauses，而这些字段/属性并不稳定可用。

2. 没有先把运行时结构确认清楚

正确顺序应该是：

  1. 先打印运行时对象类型
  2. 再确认上游 cause 的真实字段
  3. 最后写递归逻辑

但在结构尚未完全确认时，就开始写”通用递归判断”，导致方向反复漂移。

3. 验证与修复混在了一起

加了临时日志，但没有足够快地收敛假设，导致日志越来越多、逻辑分支越来越复杂，但核心问题依然是对象结构判断不稳。

4. 递归入口选得不够稳

用户真正需要的是 BuildUpstreamCause -> upstreamProject/upstreamBuild -> upstreamRun -> upstream causes，而中间尝试过多种路径（rawBuild、getCauses()、getBuildCauses()、_class、JSONObject）。尝试本身不是问题，问题在于尝试顺序没有收敛到最小可验证路径。

关键教训

教训 1：Jenkins Pipeline 里不能想当然

Pipeline / Shared Library / Sandbox 场景下，很多对象访问方式和普通 Groovy 脚本不一样：

  • rawBuild 不一定能直接用
  • _class 不一定存在
  • getBuildCauses() 返回值也不一定统一

教训 2：先确认真实类型，再写判断

对于 cause 判断，最稳妥的方式应该是：

  • 先识别真实 class
  • 再做字段读取
  • 对 net.sf.json.JSONObject 做单独兼容
  • 对真正的 Jenkins cause 对象按对象属性处理

教训 3：不要在没证据前频繁换方案

这次最大的成本不是某一行代码，而是多次切换错误假设。如果一开始就先把对象结构打清楚，整个问题会更快收敛。

正确方向

后续如果继续修这个函数，正确策略应该是：

  1. 以 currentBuild.getBuildCauses() 为入口
  2. 对 TimerTriggerCause 直接返回 true
  3. 对 BuildUpstreamCause，通过 upstreamProject 和 upstreamBuild 找到上游 run
  4. 对上游 run 再调用 getBuildCauses() 递归判断
  5. 只对 JSONObject 做必要兼容，不再假设所有对象都具有 _class

调试日志应该用于确认假设，而不是在假设未收敛时不停扩大分支。

结论

这次问题的本质并不复杂，核心需求就是：

只要构建链路任一上游是 TimerTrigger，zIsTimerTrigger() 就返回 true

没有一次做对，主要是因为：

  • 对 Jenkins 运行时对象结构判断过早
  • 递归路径设计反复切换
  • 对 rawBuild、_class、upstreamCauses 的使用顺序不够谨慎

如果重新来一次，应先用最小日志确认真实对象结构，再写最小递归逻辑，最后再去掉调试代码。这不是用户需求复杂，而是处理顺序不够严谨。
