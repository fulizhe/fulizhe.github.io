Grok 4.5 fast 踩坑实录：挤牙膏、念经与七次翻车

前言

最近在 Cursor 里挂了 Grok 4.5 fast 当编码模型用，连续踩了几个坑之后，我攒了一肚子火。不是那种”模型不够聪明”的温和失望，而是一种被当猴耍的愤怒。

这篇文章不是模型横评，没有跑 benchmark，没有对比表格。就是一次真实使用后的情绪输出和方法论反思——如果你也在考虑用 Grok 4.5 做日常编码，或者你也在用中转站接各种模型，这篇可能会帮你避几个坑。

1. 挤牙膏式实现

Grok 4.5 fast 最让人窝火的一点：它只做你字面上告诉它的那一点，做完之后还得”贴心”地告诉你，接下来可以进一步优化 xxx 相关功能。

你让它写个函数，它写完了，末尾加一句”您还可以考虑扩展错误处理和日志记录”。你照着它说的扩展了，它又写完了，再补一句”建议增加单元测试覆盖边界场景”。每一次都往前推一小步，但永远不告诉你这条路到底能走到哪。

这不是在帮你，这是在遛你。有种把人当狗溜的感觉——你永远在被它牵着走，而不是它一次性把完整方案摆在你面前。

我做的是功能测试，不是挤牙膏。你需要做的是：在第一版就把你能想到的完整功能呈现出来，然后告诉我这个方案的天花板在哪，还能往哪些方向延伸。不要像走迷宫一样，只给当下这一段路，走完才指下一个路口。

一个合格的编码助手，应该主动画出功能边界，而不是等着用户一勺一勺地喂需求。

2. 啰嗦到让人窒息

每次让 Grok 4.5 fast 分析问题，输出格式永远是一个模板：这个问题是因为 xxx，又 xxx，最后能够 xxx。

五行的代码改动，前面铺垫了五百字的”分析过程”。我得先读完这五百字，才能看到那五行代码。而读懂这五百字的时间，足够我自己把代码写完了。

这不是详尽，这是噪音。编码场景里，我对 AI 的期待是：给我代码，给我关键决策点，给我你做了什么假设。其他的，免了。

过度解释不是透明，是认知税。 每一行没有信息增量的文字，都是对用户注意力的掠夺。

DeepSeek 不这样。它给你的就是代码加一句话说清楚改了什么，干脆利落。你反而会信任它——因为它尊重你的时间。

3. 自由发挥等于灾难现场

前两个问题虽然烦，但至少还在可控范围内。真正让我崩溃的，是给它自由发挥的空间。

同一个任务：Jenkins Shared Library 里写一个 zIsTimerTrigger() 函数，递归判断构建链路的最上游根因是不是定时器触发。逻辑不复杂——沿着 BuildUpstreamCause 的 upstreamProject 和 upstreamBuild 一路追到最上游，看根因是不是 TimerTriggerCause。

DeepSeek 一把过。 代码逻辑正确，递归路径清晰，直接能用。

Grok 4.5 fast 改了七次，越改越远。 第一版只判断了当前构建的直接 cause；第二版碰 rawBuild 碰到 MissingPropertyException；第三版换成 getBuildCauses() 又错误假设返回值是统一的 Map 结构；第四版试图兼容多种对象类型，结果在 _class 属性上又炸了……七次迭代，每次都在错误的路线上修补，离正确答案越来越远。

它甚至自己写了一篇复盘，承认”过早假设了 Jenkins 返回结构”、”验证与修复混在了一起”。一个模型能写出高质量的复盘文章，却不能在第一次就做对——这本身就是一种荒诞。拿用户当猴子测试小工，在错误的路上越走越远。

能反思不代表能执行。 复盘写得再漂亮，七次试错的时间成本已经实打实地烧掉了。用户不是你的 QA，不是你的测试用例生成器，更不是你试错-修正循环里的燃料。

4. 中转站的疑云

到这里我面临一个疑问：我用的是中转站接的 Grok 4.5 fast，到底是中转站注水了，还是 Grok 4.5 fast 本身就这么废？

这不是阴谋论。中转站”注水”——即用更便宜的模型替换宣传的模型，或者降低推理参数——在国内 AI API 灰色市场是公开的秘密。你花了 Grok 4.5 的钱，跑的可能是某个不知名的小模型，这种事情并不罕见。

但我没法证实，也没法证伪。我没有 xAI 官方 API 的直连对比，也没有条件做严格的 A/B 测试。所以这个问号只能挂在这里。

我能确定的是：不管问题出在中转站还是模型本身，用户体验已经实打实地崩了。当你用了一个工具，结束后的第一反应是”我到底该怀疑渠道还是怀疑产品”——这个工具就已经失去了信任基础。

后记

写完这篇，情绪平复了不少。说到底，我对 Grok 4.5 fast 的愤怒，本质上是对”无效协作”的愤怒。一个编码助手可以不够聪明，但不能浪费你的时间；可以犯错，但不能把你当实验对象。

DeepSeek 一把过的那次经历，让我意识到：模型之间的差距，不在于谁更能写漂亮话，而在于谁更尊重用户的注意力。 五行代码加一句话，比五百字分析加七次试错，高下立判。

至于中转站的问题，后续我会想办法搞一个官方直连来验证。如果真是注水了，那这篇就是个教训——便宜的中转站，代价可能是你整个下午的时间。如果 Grok 4.5 fast 本身就是这个水平……那就让市场来说话吧。

对 cursor-grok4.5 fast 的使用感受

啰嗦。每次分析问题就是因为xxx, 又xxx，最后能够xxx，一个五行的代码工作流，文字部分写了五百字，文字理解时间比自己去手写代码长得多。
