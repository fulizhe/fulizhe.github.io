关于管理流程中的差异化对待

2026-05-08T13:50:54.000Z2026-05-13T13:10:56.733ZChina

工具的枷锁与信任的让位：反思团队过程管理与“缝隙工作”

在最近的一次关于部门效能和研发管理的探讨中，团队内就“禅道等过程管理工具的利弊”以及“领导者微观管理（Micromanagement）的边界”进行了一次复盘。这段对话触及了组织行为学中几个极其隐蔽却又极其致命的痛点：管理的颗粒度、流程与效率的博弈、信任的边界，以及隐性工作的价值。

当我们试图用冷冰冰的数据和系统来丈量一切产出时，我们或许正在亲手扼杀团队中最宝贵的自驱力。以下是我基于过往踩坑经验与组织观察，对这几个核心议题的深度展卷。

管理的底色：作为“中性词”的不信任与对抗“熵增”

在对话中，我提到了一个观点：“管理本身就代表着一种不信任，但这里的‘不信任’是一个中性词汇。”

在企业组织架构中，流程（如禅道的工时填报、任务拆解、Bug率考核）的建立，本质上是为了对抗组织的“熵增”和个体的“不确定性”。这种防守型的“不信任”，并非是在道德层面质疑员工的人品，而是在工程体系层面怀疑“产出稳定性的不确定”。

在一个成熟的商业机器里，流程设计的初衷是为了保住组织的“下限”——它确保了哪怕是能力平庸的执行者、抑或是刚入职还在熟悉环境的新人，也能在一个被高度约束的标准化框架里，产出及格的成果。“统计出来后，问题已经暴露了”，数据的精准度在这一刻展现了它的防守价值：它像一面照妖镜，能让系统性偏差无处遁形。

然而，凡事皆有代价。当这种“追求确定性的系统”向团队内最优秀的那拨人蔓延时，灾难便开始了。

流程的诅咒：对“缝隙工作”与高绩效者的反向剥削

任何一个运转良好的技术团队中，都充斥着大量无法被量化、难以被标准流程定义的“缝隙性质的工作”（业内也常称为 Glue Work / 胶水工作）。

这些工作可能是一次跨部门 API 对齐的紧急拉会，可能是帮初级开发排查了一个极其诡异的生产环境 Bug，可能是重构了一段无人认领的祖传烂代码，也可能仅仅是在两个剑拔弩张的业务线之间做了一次技术润滑。

这些不可或缺的“缝隙工作”，往往由团队里能力最强、大局观最好、责任心最重、最被信任的核心员工承担。如果管理者一刀切地用“禅道”去精细化考核这批人，必然导致两种极具破坏性的恶果：

  1. 管理内耗下的效率倒退：核心人员被迫将大量精力消耗在“把非标准工作翻译成流程语言并录入系统”上。你让他们去反向适应这些机械的流程，对于相关方来说，效率上不仅没有提升，反而是一种巨大的倒退。
  2. 劣币驱逐良币的系统性退化：当“适应流程”变得比“解决问题”更重要，当个人的绩效评定完全被系统内的数据绑架时，聪明人会迅速觉醒。他们会选择放弃吃力不讨好的“缝隙工作”，转而只接那些能体现在系统工时里、边界清晰的“标准任务”。最终，团队的协作粘性彻底丧失，系统也随之朽坏。

隐性的系统摩擦力：基于“福格行为模型”的微创新扼杀

关于为何我不愿意将所有工作（尤其是底层的缝隙工作和微小优化）全部纳入禅道等系统进行过程管理，还有一个更深层的心理学考量——福格行为模型（Fogg Behavior Model）。

福格模型指出，一个人行为的发生，需要三个要素同时具备：B = M * A * P （行为 = 动机 × 能力/简单程度 × 提示）。

在日常的研发场景中，开发人员经常会产生一些自发的“微小优化需求”。比如，发现某个高频的手动操作很繁琐，想顺手写个脚本自动化；或者发现某段代码结构不佳，想随手做个重构。

  • 在非量化的低约束环境下：开发者的动机（M）被激发，且直接动手改造的门槛极低（A），优化行为（B）就会自然发生。优化完成后，系统操作的“心智负担”降低，这又会反过来促使他们更愿意进行下一次优化，从而进入一个“低阻力 -> 微创新 -> 心智负担减轻 -> 持续优化”的正向飞轮。
  • 在强管控的过程管理（禅道）环境下：情况则截然不同。系统要求任何工作都必须先提单、填报工时、关联版本。此时，尽管开发者的动机（M）还在，但执行门槛（A，即流程带来的系统摩擦力）被无限拔高了。

当“录入系统的心智负担”大于“优化本身带来的爽感”时，开发者会怎么做？答案是：他们会悄无声息地把这些优化需求“按下去”。

为了避免繁琐的录入流程，当事人宁愿忍受平时偶尔一次的手动低效操作（因为检查频率和手动操作付出的心智总和，依然小于去走一遍禅道审批流程的痛苦）。结果就是，那些真正能提升代码质量、降低系统长期维护成本的“微创新”，在这个要求绝对透明和可控的管理流程中，被无情地扼杀了。

破局之道：情境领导力与管理中的“灰度”

这也是我当时给领导提出的核心建议：对于禅道这样的过程管理考核，必须开辟出特例。对于你信得过的、过往经受住了考验的人，应该将他们排除在严苛的过程考核之外。

在管理学中，这被称为“情境领导力（Situational Leadership）”或是“差异化管理”。管理从来不是由 0 和 1 组成的数学公式，而是一门充满“灰度”的艺术。

  • 对于低成熟度或信任尚未建立的个体：采取强管控，抓过程指标，通过数据去纠偏和辅导。
  • 对于高成熟度、久经沙场的核心战将：采取弱管控，抓核心目标与结果（ROI导向）。他们已经用一次次攻坚战役证明了自己的可靠性，管理者应该赋予他们最大的决策自由度，让他们保持敏捷的战斗身段。

方案没有绝对的好坏，只有动态的平衡。全员免除考核会导致组织失控滑向深渊，而全员死磕流程则会导致组织僵化失去灵魂。在这里，“信任”本身就是一种高级的、需要盈余积累的“管理货币”。

信任破裂的死循环：警惕“亲自下场”的微观管理陷阱

回到对话的起点，高层所面临的局面，往往会演变成一个经典的“领导下场抓管理的三步曲”：

  1. 领导看着产出不符合预期；
  2. 让下面改进，结果依然不满意；
  3. 最终只能自己亲自下场干预细节。

当走到第三步时，就像我在群里指出的，这本质上宣告了对基层团队和中层管理者的“彻底不信任”。

这是一个极其危险的“微观管理（Micromanagement）陷阱”。当领导跨越层级，直接下场干预代码细节、紧盯每一个任务状态时，表面上看是在雷厉风行地解决问题，实际上是在摧毁整个团队的自我驱动力系统。

一旦领导成为那个“兜底且做最终决定的人”，下面的人就会迅速退化为没有思想的“提线木偶”——既然无论如何你都要改、都要管，那我何必还要去主动思考？这种信任破裂带来的连环反噬，会让管理者陷入无尽的救火疲惫中，而团队的创新能力则彻底归零。

要打破这个死循环，领导层需要的不是“挽起袖子自己干”，而是退后一步去复盘系统的漏洞：是战略目标一开始就没有对齐？是评价标准过于死板？还是用错了人？

组织的进化动力：痛感倒逼与“去道德化”的防御机制

如果在前文中，我们探讨的是“僵化的流程如何伤害优秀的自驱者”，那么在另一个维度上，组织还面临着截然相反的棘手挑战：“如何面对那些在舒适圈中形成免疫系统的蛰伏团队？”

在关于禅道超期任务推送的讨论中，我以“行业部门”与“平台部门”的推行差异为例，剖析了组织进化的底层动力学。行业部门之所以在几年前就能走到前面（实现超期需求、Bug的定期精准推送），并非因为他们天生觉悟更高或更勤奋，而是因为他们直面客户与交付，“外部压力更强烈，痛感更直接”。这种直面炮火的生存危机，逼迫他们必须进化出敏捷的应对机制。

反观平台部门，由于过往产品发布范围小、被长期保护在大后方，形成了“温水煮青蛙”的效应。当经济下行、产品开始大范围部署，导致低效习惯再也藏不住时，他们不仅没有自发改进，反而试图用“历史数据庞大”、“先放宽到一个月的期限”等理由来进行妥协。这揭示了一个冷酷的管理真相：没有真正的痛感，就不会有自发的变革。环境塑造了组织的形态。

更值得深思的是，平台部门在这十几年的安逸中，已经进化出了一整套“对抗变革的生存法则”。面对外界要求其走出舒适圈的推力，整个系统会自发产生一种强大的反作用力。他们熟谙如何通过流程博弈、强调客观困难，来倒逼改革者妥协，从而将现状尽可能地重新固化为“当前这种最舒服的境地”。这是组织惯性（Organizational Inertia）最真实的写照。

面对这种阻力，管理者极易陷入“道德审判”的陷阱，认为这些老员工是“刺头”、是“恶意破坏改革”。但我认为，“你不能把这理解为一种恶意，你可以理解为一种本能，生物趋利避害的本能。”

这是一种极其关键的“去道德化”管理视角。人性追求低能量消耗与逃避痛苦，既然阻力是生物本能，那么破局的关键就不在于空洞的思想教育或激烈的清洗对抗，而是去重塑环境与激励机制——如何拆除屏障，将市场的痛感真实、无损地传导给大后方？如何让“不改变的痛苦”远远大于“改变的痛苦”？这才是推动组织跨越周期的真正发力点。

架构的隐秘法则：“按闹分配”与产品的长期主义

当我们在讨论管理流程和组织防御机制时，绝不能忽略它们对系统架构产生的深远反噬。在我们这种行政上下级约束相对较弱的团队网络中，协作往往建立在“响应力”与“技术信誉”之上。

如果你因为僵化的流程（或是平台部门的拖延本能）导致响应迟缓，系统的自组织机制就会立刻生效——简单来说，就是“人家不跟你玩了”。业务方要么自己造轮子解决，要么下次遇到类似问题直接跳过你。信任一旦在这种拉扯中消散，所谓的跨部门协同就会名存实亡，系统也会不可避免地走向孤岛化。

这直接引出了一个更深层的架构治理痛点。我们的产品类似于 Linux，是一个庞大的层次性系统：从最底层的 SDK、缝合层，到云服务后端，再到最上层的前端。理论上，面对外界的一个需求或 Bug，在任何一层都可以进行修补。并且，越是底层的干预，掌控力越强，解法也就最多，但越偏底层，其修改影响面越广。

但是，在决定“在哪层实现”之前，我们往往缺失了最致命的第一道防线：需求的源头拦截。 面对外界涌来的需求，必须有人去与需求方进行深度的沟通博弈，剥茧抽丝地挖掘其背后的“真实意图”。如果缺乏这层拦截，表面需求极易被曲解，甚至被盲目扩大化（Scope Creep），导致系统吞下大量本不该存在的伪需求。

紧接着，当这些需求进入实现阶段，如果没有一个能够概览全局的人（或机制）来审查并拍板“全局最优的实现层”，那么技术决策就会迅速退化为纯粹的职场博弈。最终的结果往往极其荒谬：不是在哪一层实现最合理，而是“哪一层的人嗓门最小、吵架输了”，那一层就去兜底做脏活。 这种缺乏全局视角的妥协解，是对工程质量的极大破坏。

更致命的是，我们做的是“产品”，而不是打一枪换一个地方的“项目”。项目可能是半年的短跑，交付即结束，不给后患发作的机会；但产品是一场马拉松，有着漫长而痛苦的维护生命周期。那些为了图快、图省事、图偷懒，或者仅仅是为了应付过程考核而草率做出的临时方案，会像“狗皮膏药”一样死死贴在系统底层。

在未来的漫长岁月里，这些狗皮膏药会不断干扰你的每一次技术决策，迫使你陷入“无限倒退兼容”的泥沼。最终，整个技术栈的演进效率会持续降低，产出无限趋近于零——这就是短视的管理与协作破裂，最终导致“技术破产”的真实路径。

后记与感悟

“三流的管理抓考勤与流程，二流的管理抓目标与数据，一流的管理抓信任与赋能。”

在研发体系的构建中，工具和流程（如禅道、Jira）是组织的骨架，它们能让庞大的团队站立不倒；但那些优秀的核心人才及其默默承担的“缝隙工作”，才是组织的肌肉和血液，决定了这支队伍能跑多快、能打多硬的仗。

真正的管理智慧，在于用流程给平庸兜底，用信任给卓越让路。

永远不要让旨在提升效率的工具，最终沦为捆绑优秀灵魂的枷锁。在技术与管理的交汇处，保留一份对人的敬畏，或许才是跳出这个死胡同的终极法则。

原始素材和基础语料（人工）

禅道这样的过程管理考核，其实应该开辟出部分人员的特例，对于你信得过的，过往经受住了你考研的，他们应该排除在考核之外。
这是符合管理的理念，否则你让这些你信得过的人，去反向适应这些流程，本身就是一种倒退。
因为很多缝隙性质的工作，都是这些人在做，如果把它们纳入禅道的管理，那对于相关方来说效率上反而是下降的
当然这也有赖于领导的权衡，方案么有绝对的好坏，只有平衡。
管理本身就代表着一种不信任。这里的不信任是一个中性词汇。

行业部门和平台部门的对比，其实我想要表达的是：

  1. 行业部门走到前面只是因为他们的外部压力更强烈，痛感更直接；这会逼着他们不得不持续思考解决方案；
  2. 平台部门就是因为过往产品发布范围小，导致人工操作，低效率习惯也能勉强跟上；而现在经济下行，加上产品逐步打开进行大范围部署，问题逐渐藏不住；但这帮人依然采用旧有习惯来应对，不愿意改进
  3. 我举例行业部门并非表扬他们，只是他们是最典型的压力倒逼改进的范例
    而平台现在很明显就站在对立面，而且更有意思的是：过去这十几年的锻炼，他们已经形成了一整套在这个过程中的生存法则
    他们也是有方法和手段，倒逼外界的努力，将现状尽可能地固化为当前这种最舒服的境地
    你也不能把这理解为一种恶意，你可以理解为一种本能，生物趋利避害的本能 。

我不愿意全部纳入禅道的另外一个原因，还是基于福格行为模型，我担心录入禅道这个工作会导致某些优化需求被当事人按下去，因为平时手动的时候，可能很长间隔一次（检查频率和需要付出的心智多重因素决定的），如果你让他录入禅道，他可能嫌麻烦就不做了；但如果优化之后让操作的心智负担减少，他就愿意多操作了，进入正向循环。
还有一点，在这个行政上下级约束很弱的团队里，你要是响应不快，人家就不跟你玩了。。。要不自己解决，要不下次就直接不回应类似的问题。
而且面对需求时候，也得有人去和需求方沟通，挖掘真实意图，避免需求被误解，被扩大化。
还有一点，我们的产品类似于liunx一样是一个层次性系统，从底层的sdk, 到缝合层， 云服务后端，再上层前端；对于外界的一个问题，其实在哪一次都能解决，而且底层肯定是解决方案更多的，因为越底层所拥有的掌控力越强；所以需要能够概览全局的人来审查需求，决定在哪里实现，实现全局最优解，否则像现在这样每次都是局部最优解，那就是只有嗓门最小，吵架输了的的那一层去做了。
而且我们做的是产品，每个决策都是会在之后漫长的维护期持续影响之后的每个决策的，而不是像项目那样最多半年就断根了，不给后患发生的机会。你图快，图省事，图偷懒而做的临时方案，会如狗皮膏药一样之后不断地干扰你之后的技术决策，无限倒退兼容，最终效率持续降低，产出持续减少接近于零，所谓的技术栈（破产）。
