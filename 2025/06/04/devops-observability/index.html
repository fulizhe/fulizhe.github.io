可观测性概述

2025-06-04T13:55:40.000Z2026-02-25T14:38:38.505ZChina

认清它，承认它，然后改变它。

前言

现状

详细的公司业务背景过往已经介绍过不少，这里不再赘述。

总而言之一句话总结就是：技术对业务的支持响应速度不足。不论是对于新增需求的完成效率，还是所发现问题的修复效率。

当然面对这样的”诋毁”，技术可以从不同的角度给出三万四千个理由去证明是要求不合理导致的。

只是这样的”理由”对于现状毫无促进，最终用户也不会因为你的这种解释而选择放弃自己的标准和要求。他们更可能的选择是让能够做到的竞争对手来接替你的工作。

当然这个时候有人会说”那正好落个清闲，不用伺候这帮大爷”，只是这种只能私下抱怨的低级言论不适合放在桌面上丢人，这里就不作进一步讨论。

作为一名合格的职场人士，面对问题，我们不仅要思考解决方案，更要积极行动。即使无法根治问题，通过技术手段进行缓解，提升客户满意度，是任何时候都应该努力实现的目标。而作为技术人员，我们更应该充分发挥专业优势，从技术角度出发，寻找切实可行的解决方案。这不仅是我们的职责所在，也是我们展现积极解决问题的态度、专业价值、赢得客户信任的关键。

分析问题

本文主要讨论”系统问题多，解决速度慢”问题的解决方案。

针对这类问题，一直都是两种思路：

  1. 确保软件系统上线后不出问题(“避错能力”)。这个思路的要求相当高，而且不现实。这从多个维度对公司提出非常高的要求：人员能力、设计先行、需求稳定少变动、足够的开发测试时间等等，总之就是得有充足的资源实现慢工出细活。公司的业务特点决定了以上这些前提没有一条是可以被满足的。
  2. 出现问题尽快解决。方案一已经被淘汰，那自然就只剩下这个方案二了。而且这也是符合我们实际情况的。我们的用户对于系统稳定性要求并不高 —— 你别太耽误我完成任务就行，至少什么业务中断，等一个小时这种状况，都行。

接下来让我们结合公司业务特点来对方案二进行详细分析。

出现问题快速解决（缩短MTTR）

首先让我们解释下”出现问题快速解决”才具有执行可行性的背景知识：

  1. 盈利是公司存在的目的。技术是成本部门，证明自己价值是技术自己的责任， 而不是什么”你业务又没要求”。
  2. 现实中，尤其是我们的业务场景，或者说绝大部分业务场景下，100%可靠的系统是不存在的。绝大部分系统都是带伤上阵的。这是VUCA时代（volatile，uncertain，complex，ambiguous）唯一的选择，只有快速响应客户需求，让客户看到可运行的实际系统，我们才能在同行竞争中取得优势，确保自己长久活下去。
  3. 生产环境千差万别，你也无法预料最终用户使用你的产品和服务的方式，不能确定他们会遇到怎么样的问题。怪用户不按照你的设想去使用软件产品？这种极其业余的言论这里就不进行驳斥了。
  4. 软件的意义就是缓解硬件的不可变，做到”ETC（Easy To Change）”。变化就意味着风险，意味着问题引入。
  5. 没有人使用的系统才没有问题。 正如C++之父Bjarne Stroustrup曾说过的一句名言：”世界上只有两种编程语言：一种是被人抱怨的，一种是没人用的”。说到这个简直闭环了，我在实际工作中确实亲眼目睹通过把问题搞得特别复杂，让使用者放弃，然后实现系统没有bug。

不能问题多，解决得还慢。把缺点全部占齐了。

实现

上面解释了为何”出现问题快速解决”才是唯一适合我们的技术路线。本小节让我们深入探讨下如何实现这个思路。

如何尽快解决？ 我们将问题解决分为两个阶段：定位问题 + 解决问题。 结合我们的业务特点，稍微有点行业经验的都有切身体会：不是难解决，而是难定位。解决问题五分钟，定位问题两小时。

正如我所一直强调的，对于传统软件企业，所谓的技术是一个”先有意识再有能力“的事儿。针对发现的问题进行长时间的观察和思考，你是一定可以成为这方面的专家（不仅是本公司，甚至行业内也是如此）。

那么，针对”难定位”的问题，经过持续性的思考实践，我们给出的解决方案正是本文标题部分所明示的：持续增强系统的可观测性。

  1. 我们一直都很好奇，为什么像我们这样的业务系统会陷入难定位？业务逻辑复杂顶多也就是if/else比较多，这和”难定位”完全不搭边。
  2. 这里面还涉及基本的”解决问题效率高”方面的认知对齐。不少研发其实还沾沾自喜，觉得自己解决问题很快。那么这里给出一个定量的标准，来方便进一步的沟通，以及诸位在平时工作中有一个对比提升的目标。这就是**”1、5、10标准”（1分钟响应，5分钟定位，10分钟解决）**。咱们以BUG修复为例，工作八小时内按照这个标准一天应该是24个BUG。当然这是理论上，那么我们拉回现实取中间值：10-12个。诸位谁做到了，或者说做到了50%+。
  3. 对于系统而言，尤其是身处一个庞大系统集群中的个体系统，可观测性的重要性只会不断增强。毕竟当出现问题时候，”尽快自证，减少噪音“是一项非常重要的能力，否则每次出现问题都卡在你这里，一两次你能够蒙蔽所有人说什么工作认真努力加班， 但次数多了自然就会成为众矢之的。
  4. 对于已经迭代了很久的软件产品，优化的改进不是一朝一夕能够解决的，如果无法做到问题的立即解决，那就先让问题能够以一种非常廉价的方式展示在尽量多的人面前。降低沟通成本，加速问题解决。
  5. “如果你不能度量它，就无法改进它”。( If you can’t measure it, you can’t improve it ) – 现代管理学之父德鲁克

福格行为模型（能用，好用，爱用）

“行为越容易完成，用户就越有可能采取行动。提高能力的关键是简化行为，让用户更容易完成”。按理极限编程的基本思想，好的最佳实践就应该做到极致，福格模型所揭示的能力/动机与行动之间的关系指引我们应该尽可能降低系统内部信息获取的难度，持续增加系统的可观测性，如此才能让所有人相信系统正在向着一个好的方向持续迭代。

持续精进

软件开发是园艺。产品迭代过程中要做到持续精进，每个问题的解决都应该让系统更加稳固，系统定位问题更快，下次即使是解决其他问题依然更快，这次十分钟，下次九分钟。保证自己每天都比昨天的自己强一点, 这类个人要求同样也是适用于我们所负责的软件产品。

很难想象一个没有技术追求的团队能开发出一个健壮的、可维护性好、可扩展性好的系统。相反，这种业务代码的堆砌，从短期看也许是“较快”实现了业务需求，但是从长远来看，这种烂系统的增加会极大地阻碍业务的发展，形成一个个的黑洞应用，而工程师被裹挟在业务需求和烂系统之间，疲于应对，心力交瘁。这种将就将导致系统腐化堕落，技术债越垒越高，丑陋的代码疯狂滋长，像肿瘤一样消耗你所有的能量，最后还要你的命。

带来的转变

  1. 成员对自己的系统有信心。这是我亲耳听到的反馈。而不是遇到问题时候的无助，那样很狼狈 —— 明明只有确定某个配置项，那眼前这个问题就确定了。

  2. 避免”过度设计”/“过度编码”。在完备”可观测性”支持下，研发人员可以放手编码，快速验证猜想并调整。而不是因为担心系统上线后难以查找问题而在前期进行大量防御性设计，宝贵精力被淹没在对于未知的恐惧中。

  3. “可观测性”让每个组员都有希望成为超级个体。很多时候，是因为获取信息的能力和熟练度导致了解决问题能力上的差异，另外一点甚至是因为没有看到这些信息导致没有意识到需要这些信息。尤其是咱们这类业务场景下。依然如此，那我就让这些信息先出现在你面前。

  4. 我们在一开始介绍的针对问题的两种方式”事前投入大量资源确保系统上线之后不出问题”和”事后进行问题的快速定位解决”，这两种方法里前者的精进对于后者的帮助有限，但是精研后者对于前者有着相当明显的促进作用，毕竟方法一里只是投入大量资源将BUG捕捉在了上线前，并不是说不会产生BUG。

  5. 错误解决越快，成本越低. 问题拖延时间是需要收利息的。难解决的问题一个特点就是怀疑的方向特别多，东一个西一个。动不动就往操作系统和硬盘怀疑。

  6. 解决问题才是我们的目的。在此之前的定位问题并不是我们的目的，为了定位问题而翻阅日志更加不是我们的目的。所以这些前置操作所消耗的时间和精力应该越少越好，最好为零。
